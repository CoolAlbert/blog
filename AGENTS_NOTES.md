# AGENTS_NOTES.md — Заметки о подагентах

## Режимы подагентов

### mode="run" (one-shot) — по умолчанию
- **Когда использовать:** Однократные задачи
  - Реализовать фичу
  - Исправить баг
  - Провести исследование
  - Сделать выгрузку данных
- **Как работает:** Подагент выполняет задачу один раз и завершается
- **Автоматически:** Отправляет отчёт о завершении в чат запросчика
- **Идеально для:** "Сделай X и сообщи результат"

### mode="session" (persistent)
- **Когда использовать:** Долгоживущие агенты
  - Мониторинг процессов
  - Периодические проверки
  - Многократное взаимодействие ( steer )
- **Как работает:** Создаёт постоянную сессию, к которой можно обращаться
- **Управление:** Через sessions_send или sessions_history
- **Идеально для:** "Следи за этим и докладывай регулярно"

## Основные принципы

### Перед запуском подагента:
1. ✅ Сам изучить issue/задачу
2. ✅ Понять, что конкретно нужно получить
3. ✅ Определить критерий успеха
4. ✅ Использовать шаблон AGENT_TASK_TEMPLATE.md

### При выборе режима:
- mode="run" → для простых задач с чётким результатом
- mode="session" → для агентов, которые будут работать долго

### Управление подагентами:
- `/subagents list` — посмотреть активные
- `/subagents kill <id>` — остановить конкретного
- `/subagents steer <id> <message>` — отправить команду в активную сессию
- `/subagents log <id>` — посмотреть логи

## Автоматические отчёты

**Важно:** Подагенты в mode="run" автоматически отправляют отчёт о завершении
- НЕ нужно создавать отдельный подагент для отправки отчёта
- Отчёт приходит как system message с результатом
- Не дублировать отчёты вручную

## Вложенность (Orchestrator Pattern)

Максимальная глубина: 5 (рекомендуется 2)

- Depth 0: Main agent — может запускать подагентов
- Depth 1: Sub-agent (orchestrator) — может запускать других если maxSpawnDepth >= 2
- Depth 2: Sub-sub-agent (worker) — НЕ может запускать других

Для включения orchestrator pattern:
```json5
{
  agents: {
    defaults: {
      subagents: {
        maxSpawnDepth: 2,
        maxChildrenPerAgent: 5,
        maxConcurrent: 8
      }
    }
  }
}
```

## Конфигурация

### Модель для подагентов:
```json5
{
  agents: {
    defaults: {
      subagents: {
        model: "zai/glm-4.7", // дефолтная модель для подагентов
        thinking: "low"
      }
    }
  }
}
```

Приоритет модели:
1. Explicit `sessions_spawn(model=...)`
2. Per-agent override `agents.list[].subagents.model`
3. Global default `agents.defaults.subagents.model`
4. Inherit from caller

## Полезные команды

### Для управления:
- `/subagents list` — список активных подагентов
- `/subagents info <id>` — метаданные (статус, время, транскрипт)
- `/subagents kill <id>` — остановить
- `/subagents steer <id> <message>` — отправить команду в активную сессию

### Для отладки:
- `/subagents log <id>` — логи выполнения
- `sessions_history(sessionKey)` — полная история сессии
- `/subagents spawn` — ручной запуск (не из агента)

## Cost Note

Каждый подагент имеет свой контекст и токены:
- Для тяжёлых задач используй более дешёвую модель для подагентов
- Main agent остаётся на качественной модели
- Настрой через `agents.defaults.subagents.model`

## Ограничения

- Подагенты не получают сессионные инструменты по умолчанию
- Session tools (sessions_*, subagents) запрещены на depth 2
- Maximum nesting depth: 5
- Max children per session: 5 (default), configurable up to 20
- Auto-archive через 60 минут после завершения

## Cleanup

- `cleanup: "delete"` — архивировать сразу после завершения
- `cleanup: "keep"` (default) — ждать auto-archive (60 мин)
- `runTimeoutSeconds` — только останавливает выполнение, НЕ архивирует

## Announce

Подагенты возвращают результат через announce step:
- Сообщение отправляется в чат запросчика
- Включает: Result, Status, Runtime, Tokens, Cost (если настроено)
- Может быть пропущено если подагент отвечает ровно "ANNOUNCE_SKIP"

## Best Practices

1. ✅ Изучай задачу перед делегированием
2. ✅ Используй mode="run" для простых задач
3. ✅ Используй mode="session" для долгоживущих агентов
4. ✅ Настрай модель подагентов для экономии токенов
5. ✅ Проверяй результаты через info/log/steer
6. ✅ Не дублируйте автоматические отчёты
7. ✅ Останавливайте неиспользуемые подагенты (/subagents kill)
